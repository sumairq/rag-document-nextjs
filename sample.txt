Retrieval-Augmented Generation: A Practical Overview

Retrieval-augmented generation, or RAG, is an architecture for building language-model applications that answer questions using a specific body of knowledge rather than relying solely on what the model memorized during training. The core idea is simple: before asking the model to answer, you first retrieve the most relevant passages from your own documents and place them in the prompt as context. The model then grounds its answer in that supplied context, and ideally cites it, so users can verify the response against the original source.

This approach addresses several well-known weaknesses of large language models. First, models have a training cutoff and cannot know about events, documents, or data created afterward. Retrieval lets you inject fresh, private, or proprietary information at query time without retraining. Second, models hallucinate: they produce fluent text that is confidently wrong. Grounding answers in retrieved passages, and showing those passages as citations, gives users a way to check the work and dramatically reduces unsupported claims. Third, fine-tuning a model on a corpus is expensive and slow, whereas indexing documents for retrieval is cheap and can be updated continuously.

A RAG system has two main phases. The first is ingestion, which happens whenever a new document arrives. During ingestion the raw file is parsed into plain text, the text is split into chunks, each chunk is converted into an embedding vector by an embedding model, and the chunks and their vectors are stored in a database. The second phase is querying. When a user asks a question, the question is embedded with the same model, the system retrieves the chunks whose vectors are closest to the question vector, those chunks are assembled into a prompt as context, and the language model generates an answer that cites the chunks it used.

The quality of a RAG system depends heavily on choices made during ingestion, especially chunking. If chunks are too large, each one mixes several unrelated ideas, so its single embedding becomes a blurry average that matches no query sharply. Large chunks also waste space in the model's limited context window and can bury the one relevant sentence among paragraphs of noise. If chunks are too small, they lose the surrounding context that makes them meaningful: a sentence containing only a pronoun or a bare statistic has nothing to anchor it, embeddings become noisy, and a complete answer may be scattered across many fragments that retrieval cannot gather together. A common middle ground is several hundred tokens per chunk, large enough to express one coherent idea yet small enough to remain a precise unit of retrieval.

Overlap between consecutive chunks is the standard remedy for blind, fixed-size splitting. Because a naive splitter cuts at arbitrary positions, it will sometimes slice a sentence, a definition, or the exact fact a user needs across a boundary. Repeating a portion of the previous chunk at the start of the next one ensures that straddling content still appears intact in at least one chunk, so retrieval can surface it. The trade-off is duplication: overlap increases the number of chunks, the storage they occupy, and the number of embedding calls, which is why overlap is usually a modest fraction of the chunk size rather than half of it.

Retrieval itself is typically powered by approximate nearest-neighbor search over the embedding vectors. Exact search compares the query against every stored vector and becomes slow as the corpus grows, so production systems use index structures such as HNSW or IVFFlat that trade a small amount of recall for a large gain in speed. Cosine distance is the most common similarity measure for text embeddings because it compares the direction of vectors while ignoring their magnitude, which aligns well with how semantic meaning is encoded.

Beyond plain vector search, mature systems layer on additional techniques. Hybrid search combines vector similarity with traditional keyword search so that exact terms, names, and identifiers are not missed. Re-ranking takes a larger candidate set and reorders it with a more expensive but more accurate model. Query rewriting reformulates the user's question to improve recall. Metadata filtering restricts retrieval to a particular document, author, or date range. Each of these adds engineering complexity, and together they are where most of the effort in a serious RAG deployment is spent. The embedding and generation calls, by contrast, are comparatively thin wrappers around a model API.

Citations are what turn a RAG answer from a plausible paragraph into a trustworthy one. To support them, each stored chunk must retain enough metadata to point back at its origin: which document it came from, its position within that document, and ideally the page number and character offsets of the original span. With that information the application can render a citation that links directly to the exact location in the source, letting a skeptical user confirm that the answer is actually supported by the documents they uploaded.
